# Руководство по лучшим практикам Vue

Используйте этот навык как набор инструкций. Следуйте рабочему процессу по порядку, если
пользователь явно не запросил другой порядок.

## Основные принципы
- **Поддерживайте предсказуемость состояния:** один источник истины, все остальное выводите из него.
- **Делайте поток данных явным:** В основном, свойства (Props) вниз, события (Events) вверх.
- **Отдавайте предпочтение небольшим, сфокусированным компонентам:** их легче тестировать,
переиспользовать и поддерживать.
- **Избегайте ненужных повторных рендеров:** разумно используйте вычисляемые свойства и наблюдатели.
- **Читаемость имеет значение:** пишите чистый, самодокументируемый код.

## 1) Подтверждение архитектуры перед кодированием (обязательно)

- Стандартный стек: Vue 3 + Composition API + `<script setup lang="ts">`.

### 1.1 Основные справочные материалы для обязательного прочтения (обязательно)

- Перед реализацией любой задачи на Vue убедитесь, что вы прочитали и применяете эти основные
справочные материалы:
  - `references/reactivity.md`
  - `references/sfc.md`
  - `references/component-data-flow.md`
  - `references/composables.md`
- Держите эти справочные материалы в активном рабочем контексте на протяжении всей задачи, а не
только тогда, когда возникает конкретная проблема.

### 1.2 Планирование границ компонентов перед кодированием (обязательно)

Create a brief component map before implementation for any non-trivial feature.

- Определите единственную ответственность каждого компонента в одном предложении.
- По умолчанию оставляйте корневые компоненты и компоненты представлений на уровне маршрутов в
качестве поверхностей для композиции.
- Выносите функциональность UI и логику функций из корневых компонентов/компонентов представлений,
если только задача не является intentionally tiny single-file demo.
- Определите контракты свойств (props) и событий (emits) для каждого дочернего компонента на карте.
- Отдавайте предпочтению структуре папок по функциям (`components/<feature>/...`, `composables/use<Feature>.ts`)
при добавлении более чем одного компонента.

## 2) Применение основополагающих принципов Vue (обязательно)

Это основополагающие, обязательные к знанию принципы. Применяйте все их в каждой задаче на Vue,
используя основные справочные материалы, уже загруженные в разделе `1.1`.

### Реактивность

- Обязательный справочный материал из `1.1`: [reactivity](references/reactivity.md)
- Держите исходное состояние минимальным (`ref`/`reactive`), выводите все возможное с помощью `computed`.
- Используйте наблюдатели (watchers) для побочных эффектов, если это необходимо.
- Избегайте пересчета дорогостоящей логики в шаблонах.

### Структура SFC и безопасность шаблонов

- Обязательный справочный материал из `1.1`: [sfc](references/sfc.md)
- Поддерживайте порядок разделов SFC: `<script>` → `<template>` → `<style>`.
- Поддерживайте сфокусированность ответственности SFC; разделяйте большие компоненты.
- Делайте шаблоны декларативными; переносите ветвления/вычисления в скрипт.
- Применяйте правила безопасности шаблонов Vue (`v-html`, рендеринг списков, выбор условного рендеринга).

### Держите компоненты сфокусированными

Разделите компонент, когда у него **появляется более одной явной ответственности** (например,
оркестрация данных + UI или несколько независимых частей UI).

- Отдавайте предпочтение **более мелким компонентам + композаблам (composables)**, а не одному
"мега-компоненту".
- Выносите **части UI** в дочерние компоненты (свойства на вход, события на выход).
- Выносите **состояние/побочные эффекты** в композаблы (`useXxx()`).

Применяйте объективные критерии для разделения. Разделите компонент, если выполняется **любое** из
следующих условий:

- Он одновременно управляет оркестрацией/состоянием и содержит значительную разметку для нескольких
разделов.
- У него есть 3+ различных раздела UI (например: форма, фильтры, список, футер/статус).
- Блок шаблона повторяется или может стать переиспользуемым (строки элементов, карточки, записи в
списке).

Правило для корневых компонентов и компонентов представлений на уровне маршрутов:

- Оставляйте корневые компоненты и компоненты представлений на уровне маршрутов "тонкими":
оболочка/макет приложения, подключение провайдеров и композиция функций.
- Не размещайте полные реализации функций в корневых компонентах/компонентах представлений, когда
эти функции содержат независимые части.
- Для функций CRUD/списков (todo, таблица, каталог, входящие) разделите как минимум на:
  - компонент-контейнер функции
  - компонент ввода/формы
  - компонент списка (и/или элемента)
  - компонент футера/действий или фильтра/статуса
- Допускайте реализацию в одном файле только для очень маленьких демонстрационных примеров; если
выбран этот вариант, явно обоснуйте, почему разделение не требуется.

### Поток данных в компонентах

- Обязательный справочный материал из `1.1`: [component-data-flow](references/component-data-flow.md)
- Используйте свойства вниз, события вверх в качестве основной модели.
- Используйте `v-model` только для настоящих двусторонних контрактов компонентов.
- Используйте `provide/inject` только для глубоких зависимостей в дереве или общего контекста.
- Делайте контракты явными и типизированными с помощью `defineProps`, `defineEmits` и `InjectionKey`
при необходимости.

### Композаблы (Composables)

- Обязательный справочный материал из `1.1`: [composables](references/composables.md)
- Выносите логику в композаблы, когда она переиспользуется, содержит состояние или насыщена
побочными эффектами.
- Делайте API композаблов небольшими, типизированными и предсказуемыми.
- Отделяйте логику функций от презентационных компонентов.

## 3) Рассмотрение дополнительных функций только в соответствии с требованиями

### 3.1 Стандартные дополнительные функции

Не добавляйте их по умолчанию. Загружайте соответствующий справочный материал только тогда, когда 
существует требование.

- Слоты (Slots): родителю нужно контролировать содержимое/макет дочернего элемента -> [component-slots](references/component-slots.md)
- Сквозные атрибуты (Fallthrough attributes): компоненты-обертки/базовые компоненты должны
безопасно передавать атрибуты/события -> [component-fallthrough-attrs](references/component-fallthrough-attrs.md)
- Встроенный компонент `<KeepAlive>` для кэширования состояний представлений -> [component-keep-alive](references/component-keep-alive.md)
- Встроенный компонент `<Teleport>` для оверлеев/порталов -> [component-teleport](references/component-teleport.md)
- Встроенный компонент `<Suspense>` для границ отката асинхронного поддерева -> [component-suspense](references/component-suspense.md)
- Функции анимации: выберите самый простой подход, соответствующий требуемому поведению движения.
  - Встроенный компонент `<Transition>` для эффектов появления/исчезновения -> [transition](references/component-transition.md)
  - Встроенный компонент `<TransitionGroup>` для анимированных мутаций списка -> [transition-group](references/component-transition-group.md)
  - Анимация на основе классов для эффектов, не связанных с появлением/исчезновением -> [animation-class-based-technique](references/animation-class-based-technique.md)
  - Анимация на основе состояния для анимации, управляемой пользовательским вводом -> [animation-state-driven-technique](references/animation-state-driven-technique.md)

### 3.2 Менее распространенные дополнительные функции

Используйте их только при наличии явной потребности продукта или технической необходимости.

- Директивы: поведение специфично для DOM и не подходит для композабла/компонента -> [directives](references/directives.md)
- Асинхронные компоненты: тяжелый/редко используемый UI следует подгружать лениво -> [component-async](references/component-async.md)
- Плагины, когда поведение должно быть установлено на уровне всего приложения -> [plugins](references/plugins.md)
- Паттерны управления состоянием: общее для всего приложения состояние, пересекающее границы функций -> [state-management](references/state-management.md)

## 4) Выполнение оптимизации производительности после того, как поведение станет корректным

Работа над производительностью — это этап после обеспечения функциональности. Не оптимизируйте до
тех пор, пока основное поведение не будет реализовано и проверено.

- Узкие места рендеринга больших списков -> [perf-virtualize-large-lists](references/perf-virtualize-large-lists.md)
- Статические поддеревья, перерендеривающиеся ненужно -> [perf-v-once-v-memo-directives](references/perf-v-once-v-memo-directives.md)
- Чрезмерная абстракция в критически важных путях списков -> [perf-avoid-component-abstraction-in-lists](references/perf-avoid-component-abstraction-in-lists.md)
- Дорогостоящие обновления, срабатывающие слишком часто -> [peft-updated-hook.md](references/peft-updated-hook.md)

## 5) Финальная самопроверка перед завершением

- Основное поведение работает и соответствует требованиям.
- Все обязательные справочные материалы были прочитаны и применены.
- Модель реактивности минимальна и предсказуема.
- Структура SFC и правила шаблонов соблюдены.
- Компоненты сфокусированы и хорошо структурированы, разделяются при необходимости.
- Корневые компоненты и компоненты представлений на уровне маршрутов остаются поверхностями для
композиции, если только нет явного исключения для небольшого демо.
- Решения о разделении компонентов явны и обоснованы (границы ответственности четкие).
- Контракты потока данных явны и типизированы.
- Композаблы используются там, где переиспользование/сложность оправдывают их.
- Состояние/побочные эффекты вынесены в композаблы, если это применимо.
- Дополнительные функции используются только тогда, когда этого требуют требования.
- Изменения для производительности применялись только после завершения функциональности.
